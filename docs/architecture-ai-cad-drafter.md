# Architecture: Closing the AI CAD Drafter Loop

**Goal:** let the user talk to the chat agent ("Drawing Assistant") and have
it actually change the generated drawing — materials, visibility, and
merging sections — the same way dragging a slider already does for
dimensions. Today only dimensions are wired end-to-end; see
`docs/gap-log.md` for the specific gaps this closes.

This doc is written for a coding agent with no memory of how we got here.
Read `docs/gap-log.md` first for the "what's broken" framing; this doc is
the "how to fix it."

## Current architecture (context you need)

- **Backend**: FastAPI app in `backend-python/app/`. Furniture geometry is
  generated by per-type builder functions: `dxf_exporter.py` (`save_round_pedestal_table`,
  `save_rectangular_table`, etc. — produces the real downloadable DXF) and
  `drawing_model.py` (`build_round_pedestal_model`, etc. — produces an
  intermediate `DrawingModel` object that `svg_exporter.py` renders to the
  inline SVG preview shown in the UI). **These two paths are independent
  implementations of the same drawing** — they must be kept in sync by hand
  whenever you change one (this has already caused bugs; see gap-log history
  in git log for "collar plate" and "cone" fixes).
- **`_component_schema(f_type)`** in `routes.py` (~line 97) returns a list of
  `{name, label, dims: [{key, label, min, max, step, unit}]}` per furniture
  type. This is what the frontend's `SliderPanel.tsx` renders as grouped,
  editable sections, and what `svg_exporter.py`'s `data-component="{name}"`
  hit-areas let the user click on. **Component names here are the contract**
  — they must match: (a) `PolygonComponent.name` in `drawing_model.py`, (b)
  the `data-component` attribute `svg_exporter.py` emits, (c) section names
  in `_component_schema`.
- **`/adjust`** (`routes.py`, `adjust_dimensions`) accepts dimension form
  fields by name (`top_diameter_cm`, `base_diameter_cm`, `collar_diameter_cm`,
  etc.), regenerates the DXF + SVG, and returns the resolved dimensions. This
  is the ONLY mutation endpoint right now — it has no concept of materials,
  visibility, or merging.
- **`chat_agent.py`**: `process_message()` calls GPT-4o (or local Ollama
  fallback) with a static `SYSTEM_PROMPT`, gets back `{response, action,
  updates: {furniture_type, materials, dimensions, visibility, notes}}`,
  and merges `updates` into a `DrawingState` dataclass (materials/dimensions/
  visibility/notes/corrections). `/chat` in `routes.py` persists this state
  in an in-memory `CHAT_SESSIONS` dict keyed by `session_id`.
- **`ChatBox.tsx`**: posts to `/chat`, and ONLY when the response includes
  `action: "render"` AND `state.dimensions` is non-empty, separately posts
  those dimensions to `/adjust`. Materials/visibility from chat currently go
  nowhere.

## Design

### 1. Materials — give them a real effect

**Data model.** Add a `materials: Optional[Dict[str, str]]` parameter to
every builder function (`save_round_pedestal_table`, `build_round_pedestal_model`,
and eventually the other 7 templates), keyed by component name (same names
as `_component_schema`). Default `None` → current hardcoded behavior
(no change for existing callers).

**Effect of a material override, in priority order:**
1. **Leader annotation text** — instead of the hardcoded string (e.g.
   `drawing_model.py:493`'s `"Black hammered textured- apply a layer of PU
   coating for paint protection"`), interpolate the user's material
   description into a template: `f"{material.capitalize()} — {component_default_finish_note}"`.
   Keep the existing finish/coating notes as a suffix so material changes
   don't lose the "apply PU coating" type instructions — just swap the
   material name itself.
2. **Hatch pattern + color** — add a small keyword→hatch lookup (e.g.
   `wood`/`oak`/`walnut` → `ANSI31` wood-grain pattern (already used for
   tabletop); `steel`/`metal`/`brass`/`brushed` → `ANSI37` or a new
   metal-look pattern; `marble`/`stone` → a denser cross-hatch; fallback to
   the component's existing default pattern if no keyword matches). This is
   a heuristic, not NLP — keep it simple and obviously extensible (a dict of
   `{keyword: (pattern, color)}`).

**API contract.** Add `materials_json: str = Form(None)` to `/adjust` (a
JSON-encoded `{component_name: material_description}` dict — using one JSON
field instead of N separate form fields because the set of components
varies per furniture type, unlike the fixed dimension keys). Parse it,
merge with any existing materials state for that DXF (read from the
`.json` sidecar file `_save_drawing_model` already writes — see
`routes.py:30`), and pass through to both builder calls.

**Component schema extension.** Add a `"material"` key per schema section
in `_component_schema()`, e.g.:
```python
{"name": "pedestal_body", "label": "Pedestal Column", "dims": [...],
 "material": {"key": "pedestal_body", "default": "Black hammered textured steel"}}
```
Frontend (`SliderPanel.tsx`) renders a text input per section using this
(parallel to the existing number input pattern) — out of scope detail for
this doc, but the schema needs the field so the frontend has something to
render.

### 2. Visibility — wire the existing mechanism to user intent

Good news: the hide/dash mechanism already exists
(`_visible()`/`_is_estimated()` in `dxf_exporter.py:240-247`, driven by
`anti_hallucination_validator.py`'s confidence scoring). It just has no
input from the user.

**Change:** thread an optional `force_visibility: Optional[Dict[str, bool]]`
param into `save_round_pedestal_table` (and friends). Change `_visible(name)`
to check `force_visibility` FIRST (if the user explicitly set True/False for
this component, that wins outright — no dashing, no confidence math), and
only fall through to the confidence-based `vr.components` check if the user
hasn't said anything about that component.

**API contract.** Add `visibility_json: str = Form(None)` to `/adjust`,
same JSON-dict pattern as materials.

### 3. Merge sections — the big one, needs a real refactor

This is qualitatively different from #1/#2: there is currently no
intermediate representation of "a furniture template is a vertical stack of
named segments" — each builder function draws hardcoded absolute-coordinate
polygons directly. You cannot "merge" two polygons that were never expressed
as separate, removable segments in the first place without restructuring
how geometry is built.

**Recommended approach — introduce a `ColumnSegment` intermediate type:**

```python
@dataclass
class ColumnSegment:
    name: str            # matches component schema name
    top_width_cm: float
    bottom_width_cm: float
    height_cm: float
    material: Optional[str] = None
    hatch: Optional[str] = None
    leader_text: Optional[str] = None
```

Refactor `build_round_pedestal_model`'s front-view logic (and eventually
`save_round_pedestal_table`'s) to first build a `List[ColumnSegment]` top to
bottom (tabletop, collar_plate, neck_ring, pedestal_body, base_plate), THEN
have a single shared function `_segments_to_polygons(segments, merge_groups)`
that:
1. Walks the list accumulating absolute Y positions (this is just the
   existing height-distribution math, moved into one place instead of
   duplicated across `dxf_exporter.py` and `drawing_model.py` — a welcome
   side effect: fixes the "two independent implementations can drift" issue
   called out above).
2. For any consecutive segments whose names both appear in the same
   `merge_groups` entry (a `List[List[str]]`, e.g. `[["neck_ring",
   "pedestal_body"]]`), **skip drawing the internal boundary line** between
   them and emit ONE polygon spanning from the first segment's top to the
   last segment's bottom, using a single linear taper from
   `first.top_width_cm` to `last.bottom_width_cm` (a trapezoid across the
   whole merged span, rather than the original piecewise shape — this is the
   simplest visually-correct interpretation of "treat these as one piece").
   The merged polygon's name becomes a composite (e.g. `"neck_ring+pedestal_body"`)
   and its leader text/material default to whichever segment the user
   specified a material for, or the first segment's if neither was specified.
3. Returns the same `PolygonComponent`/DXF-polyline list the existing code
   already produces, so downstream rendering (dimensions, leaders, hatches)
   doesn't need to change.

**API contract.** Add `merge_groups_json: str = Form(None)` to `/adjust` —
JSON-encoded `List[List[str]]`.

**Sequencing note:** do this AFTER #1 and #2 land and are stable. It touches
shared geometry math in both `dxf_exporter.py` and `drawing_model.py` and is
meaningfully riskier (regression risk on existing working shapes) than the
additive, default-`None`-preserves-old-behavior changes in #1/#2. Get a
regression test in `backend-python/tests/` asserting entity counts/positions
for the unmerged case BEFORE refactoring, so you can tell quickly if the
refactor changed unmerged output.

### 4. Make the chat agent's vocabulary dynamic

**Change `chat_agent.process_message()`'s signature** to accept an optional
`component_schema: Optional[list]` and `furniture_type: Optional[str]`.
When present, build the `SYSTEM_PROMPT`'s component/dimension lists from the
schema instead of the hardcoded lists at `chat_agent.py:89-90`:

```python
def _schema_to_prompt_fragment(schema: list) -> str:
    lines = []
    for section in schema:
        dim_keys = ", ".join(d["key"] for d in section["dims"])
        lines.append(f'- {section["name"]} ("{section["label"]}"): dims [{dim_keys}]')
    return "\n".join(lines)
```

Interpolate this into the prompt in place of the static lists. Also add
`"merge": [["component_a", "component_b"]]` to the JSON response schema the
LLM is asked to return, and parse it into `DrawingState.merge_groups:
List[List[str]] = field(default_factory=list)` (new field).

**`/chat` route** (`routes.py:1197`) needs to know the current
`furniture_type` for the session to fetch its schema — add a
`furniture_type: str = Form(None)` param (frontend already knows this from
`cadEngineResult.furniture.type`) and call `_component_schema(furniture_type)`
before invoking `chat_with_agent`.

### 5. ChatBox forwards everything, not just dimensions

**`ChatBox.tsx`'s `sendMessage`** — after `data.action === 'render'`,
currently only POSTs `/adjust` with `data.state.dimensions`. Extend to also
include (when non-empty): `materials_json: JSON.stringify(data.state.materials)`,
`visibility_json: JSON.stringify(data.state.visibility)`,
`merge_groups_json: JSON.stringify(data.state.merge_groups)`. Also stop
showing "drawing updated" (`ChatBox.tsx:145-147`) unless at least one of
these four fields was actually non-empty — right now it claims success even
when nothing changed (see gap-log #5).

## Suggested implementation order

1. **#4 first, alone** (dynamic chat vocabulary) — lowest risk, no rendering
   changes, just keeps the chat's knowledge in sync with what
   `_component_schema` already exposes. Quick win, unblocks testing the
   others conversationally.
2. **#2 (visibility)** — small, additive, reuses an existing mechanism.
3. **#1 (materials)** — additive, moderate size (hatch/text template work).
4. **#5 (ChatBox forwarding)** — do this alongside #1/#2 so they're actually
   reachable from the chat UI, not just from a direct API call.
5. **#3 (merge) last** — the only one requiring a real geometry refactor;
   do it once the contract (`merge_groups_json`) and conventions from #1/#2
   are proven out.

## Testing notes

- `backend-python/tests/test_dxf_exporter.py` and `test_regression.py` are
  the existing smoke tests (entity-count assertions per template). Extend
  them with cases for `force_visibility={"base_plate": False}` (assert the
  base_plate polyline is absent) and a materials case (assert the leader
  text contains the material string) before touching merge logic.
- Verify both render paths stay in sync after each change: generate a DXF
  via `save_round_pedestal_table(...)` and an SVG via
  `build_round_pedestal_model(...)` with identical params, and diff their
  resolved component dimensions/positions.
